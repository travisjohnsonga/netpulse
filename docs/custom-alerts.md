# Custom Alerts & InfluxDB Metrics

## Overview

spane stores time-series metrics in **InfluxDB 2.7** (org `netpulse`, bucket
`metrics` by default). Alerts are not generated by a generic "evaluate any
metric against any threshold" engine — instead, each background engine
(stream-processor, reachability monitor, check engine, environment poll, circuit
checks, config manager) emits a **named alert** that is tied to a seeded
`AlertRule` row. The rule controls the alert's **severity**, **cooldown**, and
whether it fires at all (toggle it off to suppress that class of alert).

This page documents the real InfluxDB schema (measurements / fields / tags), the
built-in alert rules, how alert rules actually work, and how to query the metrics
directly.

> **Important:** there is **no free-text expression language** (e.g. `cpu_pct >
> 90`) on the alert rule. Most thresholds are configured elsewhere — per-circuit
> on the WAN circuit, or via environment variables for temperature/PoE — not on
> the rule. See [How Alert Rules Work](#how-alert-rules-work) before assuming a
> custom condition will generate alerts.

## Available Measurements

These are the measurements actually written to InfluxDB. (Flow records are an
exception — see [the flow note](#flow-records-opensearch-not-influxdb).)

### `telemetry`

Device-level scalar metrics derived from each SNMP/gNMI poll by the
stream-processor. Default SNMP poll interval is **60s** (per polling profile);
gNMI streams arrive continuously.

- **Tags:** `device_id`, `protocol` (`snmp` / `gnmi` / …)
- **Fields (when available for the platform):** `cpu_pct` (%), `memory_used_pct`
  (%), `memory_used_bytes`, `memory_total_bytes`, `temp_max_c`, `fan_count`,
  `psu_count`, `poll_duration_ms`, plus the raw per-poll counter/gauge values.

> Field availability is platform-dependent — e.g. some platforms expose
> `cpu_pct` but no memory, others the reverse. Don't assume every field is
> present on every device.

### `interface_stats`

Per-interface rates, derived by the stream-processor from successive counter
samples (so a field only appears once there are at least two samples to delta).

- **Tags:** `device_id`, `if_index` (numeric ifIndex for SNMP), `if_name` (only
  for gNMI, where the bucket key is the interface name)
- **Fields:** `in_bps`, `out_bps` (bits/sec), `in_pps`, `out_pps`,
  `in_errors_rate`, `out_errors_rate`, `in_discards_rate`, `out_discards_rate`,
  `in_util_pct`, `out_util_pct` (% of link capacity, only when interface speed is
  known), `oper_status` (1 = up).

> Rates are stored in **bits per second** (`in_bps` / `out_bps`), not Mbps. The
> WAN-circuit utilization endpoint reads `in_bps`/`out_bps` from this measurement
> and converts to Mbps for display.

### `device_environment`

Environmental sensors. Written from two sources, sharing the same schema:
- the stream-processor, derived from SNMP ENTITY-SENSOR/hrStorage data on each poll;
- the scheduler's **environment/PoE poll** (AOS-CX REST), default every **5 minutes**
  (`ENVIRONMENT_POLL_INTERVAL_S`), so the data is stored for trending/alerting
  even when the device's Environment tab is closed.

One point is written per sensor; the `sensor_type` tag distinguishes them.

- **Tags:** `device_id`, `sensor_name`, `sensor_type`
  (`temperature` / `fan` / `psu` / `poe`)
- **Temperature fields** (`sensor_type=temperature`): `temperature_c` (°C),
  `status_ok` (1/0)
- **Fan fields** (`sensor_type=fan`): `fan_rpm` (RPM, `-1` when unknown),
  `status` (`ok` / `fault`)
- **PSU fields** (`sensor_type=psu`): `watts` (`-1` when unknown), `status`
  (`online` / `offline`)
- **PoE fields** (`sensor_type=poe`, `sensor_name=poe`): `poe_budget_watts`,
  `poe_used_watts`, `poe_used_pct` (when available), `poe_status`

### `device_reachability`

Reachability + RTT history, written by the reachability monitor. It TCP-connects
to port 22 then falls back to 443. Default interval is **30s**
(`run_reachability_monitor --interval 30`).

- **Tags:** `device_id`, `hostname`
- **Fields:** `is_reachable` (1/0), `rtt_ms` (only written when reachable)

### `transit_latency`

Path/transit latency observations (from flow-derived latency records).

- **Tags:** `src_device`, `dst_device`, `ip_protocol`
- **Fields:** `latency_ms`

### Flow records (OpenSearch, not InfluxDB)

NetFlow / IPFIX / sFlow flow records are **not** stored in InfluxDB. The
stream-processor indexes them into **OpenSearch** (index pattern
`netpulse-flows-*`) for the Flow Analytics page. A high-volume flow triggers the
`flow-threshold-exceeded` alert (throughput = `bytes / duration`, floored to a
sane minimum duration). The only flow-adjacent InfluxDB measurement is
`transit_latency` above.

Other agent-sourced measurements (`cpu`, `memory`, `disk`, `load`, `interface`)
exist for the spane Agent (server monitoring) and are out of scope here.

## Built-in Alert Rules

These are seeded by `python manage.py seed_alert_rules` (run automatically on api
startup). The **Name** column is the actual `AlertRule.name` — the engines match
their emitted alerts to the rule by this name, so it must match exactly. Rules
are marked `is_system` (cannot be deleted; disable with the toggle instead).
Severity is the seeded default; you can change it in the UI.

| Rule (name) | Source / trigger | Default severity | Cooldown |
|---|---|---|---|
| `Interface State Change` | Monitored interface up/down transition | high | 0m |
| `device-unreachable` | Failed TCP/22 (then 443) reachability check | critical | 60m |
| `service-check-failed` | An agentless service check is down/degraded | high | 60m |
| `flow-threshold-exceeded` | NetFlow/sFlow volume over the configured threshold | high | 60m |
| `latency-threshold-exceeded` | Measured path latency over threshold | medium | 60m |
| `log-anomaly-detected` | Syslog/trap matched an anomaly keyword | medium | 60m |
| `High Ping Latency` | RTT over warning threshold (default >100ms for 3 checks) | medium | 30m |
| `Ping Latency Critical` | RTT over critical threshold (default >500ms for 2 checks) | high | 15m |
| `Config Changed` | Scheduled config collection found a running-config change | medium | 0m |
| `High Temperature Warning` | Temp sensor ≥ warning threshold (default 75°C, `TEMP_WARNING_C`) | medium | 30m |
| `High Temperature Critical` | Temp sensor ≥ critical threshold (default 85°C, `TEMP_CRITICAL_C`) | critical | 15m |
| `Temperature Sensor Failed` | Temp sensor reported a non-operational status | high | 60m |
| `High PoE Usage` | PoE usage ≥ % of budget (default 80%, `POE_ALERT_THRESHOLD_PCT`) | medium | 0m |
| `High WAN Utilization` | WAN circuit over its per-circuit `alert_threshold_pct` | medium | 0m |
| `WAN Contract Expiring` | WAN circuit contract end date within 90/60/30/14/7 days | medium | 0m |

> Some alerts you'll see in the UI are emitted by an engine against a rule that
> is **auto-created on first fire** rather than pre-seeded (e.g. the "Startup
> config not saved" / `config_unsaved` alert from the compliance reconciler).
> The seeder adopts any such pre-existing rule as a system rule the next time it
> runs.

## How Alert Rules Work

The `AlertRule` model has these fields:

- `name` — must match what an engine emits.
- `severity` — `critical` / `high` / `medium` / `low` / `info`.
- `condition` — a free-form **JSON** descriptor (e.g.
  `{"source": "circuits", "metric": "wan_utilization"}`). It documents what the
  rule represents; it is **not** evaluated as a generic expression against
  arbitrary InfluxDB metrics.
- `cooldown_minutes` — minimum gap between repeat alerts.
- `is_active` — toggle off to suppress that alert class.
- `is_system` — seeded/protected (set only by the seeder).

When an engine raises an alert, it creates an `AlertEvent` linked to the matching
rule. Per-event detail lives in the event's `labels` and `annotations` JSON
(e.g. `title`, `message`, `device`, `interface`, `severity`), which the engine
fills in directly — there is **no `{hostname}` / `{cpu_pct}` template string** on
the rule. Interface alerts even carry a per-event severity in `annotations`
that overrides the rule's default.

### Creating / editing rules (Settings → Alerting → Alert Rules)

In the UI: **Settings → Alerting → Alert Rules**. You can:

- **Toggle** any rule on/off (disabling a system rule suppresses its alerts).
- **Change** severity and cooldown.
- **Add Rule** — sets `name`, `severity`, `cooldown_minutes`, and a raw JSON
  `condition` (the modal pre-fills `{"metric": "cpu", "op": ">", "threshold":
  90}`).

> ⚠️ **Adding a brand-new rule does not, by itself, make alerts appear.** Because
> there is no generic metric evaluator, a custom rule only produces events if a
> background engine emits an alert whose `rule_name` matches your rule's `name`.
> In practice, the rules you'll actually tune are the **built-in** ones above
> (severity, cooldown, on/off). To change a threshold, change it at its real
> source: the WAN circuit's `alert_threshold_pct`, or the `TEMP_WARNING_C` /
> `TEMP_CRITICAL_C` / `POE_ALERT_THRESHOLD_PCT` environment variables.

## Querying InfluxDB Directly

InfluxDB is an **internal-only** service on the `netpulse-net` bridge — it does
**not** publish a host port (unlike OpenSearch, which is bound to
`127.0.0.1:9200`). You therefore cannot reach it at `http://your-server:8086`
from outside the stack. Query it from inside a container on the bridge, e.g. the
api container:

```bash
# Open a shell on the api container
docker compose exec api bash

# Query with the influx CLI from another container, or use curl to the
# in-network hostname http://influxdb:8086 with the admin token from .env
# (INFLUXDB_ADMIN_TOKEN). Org = netpulse, bucket = metrics by default.
curl -s http://influxdb:8086/api/v2/query?org=netpulse \
  -H "Authorization: Token $INFLUXDB_ADMIN_TOKEN" \
  -H "Accept: application/csv" \
  -H "Content-Type: application/vnd.flux" \
  -d 'from(bucket:"metrics")
        |> range(start:-1h)
        |> filter(fn:(r)=> r._measurement=="interface_stats" and r._field=="in_bps")
        |> last()'
```

Adjust `org` / bucket if you overrode `INFLUXDB_ORG` / `INFLUXDB_BUCKET` in
`.env`. The admin token is `INFLUXDB_ADMIN_TOKEN` (the api reads it as
`INFLUXDB_TOKEN`). If you want the InfluxDB web UI, temporarily publish the port
by adding `ports: ["127.0.0.1:8086:8086"]` to the `influxdb` service in a
`docker-compose.override.yml` — but treat it as a debugging step, not a
production exposure.

## Checking Metric Collection

- **UI:** a device's **Collection Status** (on the device detail page) shows the
  last poll, protocol, and whether SNMP is suppressed by adaptive polling. The
  **Config Health** panel (Settings → Compliance) covers config collection.
- **CLI:** `./netpulse.sh status` (or `docker compose ps`) shows which services
  are up.
- **Last-seen via the shell:**

```bash
docker compose exec api python manage.py shell -c "
from apps.devices.models import Device
for d in Device.objects.all()[:20]:
    print(d.hostname, d.last_seen, d.is_reachable)
"
```

(`Device.last_seen` is the device's last-seen timestamp; `is_reachable` /
`unreachable_since` track reachability state.)

## Metric Retention

> spane does **not** ship a configurable InfluxDB retention setting or a
> "Settings → System → Storage" page. InfluxDB keeps points per the bucket's own
> retention (the default bucket created at install is effectively infinite unless
> you set one). The numbers below are **recommendations** you can apply yourself
> by setting a retention period on the `metrics` bucket (or by using separate
> buckets per measurement) — they are not enforced by the app.

| Data | Suggested retention |
|---|---|
| `interface_stats` | 90 days |
| `telemetry` (cpu/mem) | 90 days |
| `device_environment` | 365 days (long-term trending) |
| `device_reachability` | 30 days |
| `transit_latency` | 30 days |
| Flows (OpenSearch `netpulse-flows-*`) | 30 days (manage in OpenSearch, not InfluxDB) |

The only retention that **is** enforced in-app is for relational data:
resolved `AlertEvent`s are purged after ~90 days (`purge_resolved_alerts`), and
audit-log retention defaults to 90 days (`AUDIT_LOG_RETENTION_DAYS`). Neither
affects InfluxDB.
